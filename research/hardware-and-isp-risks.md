# Hardware и ISP риски

Этот документ фиксирует риски, которые могут помешать масштабированию продукта с преднастроенными VPN-роутерами. Backend сам по себе не решает эти проблемы: совместимость железа, прошивки и интернет-провайдера является продуктовым риском.

## Правило работы с внешними материалами

Сторонние прошивки, скрипты, пакеты, бинарные файлы, изображения, тексты, конфигурации и другие assets можно использовать как источники знаний и ссылок, но нельзя копировать в репозиторий, пока не проверены лицензия, условия использования и явное разрешение на хранение или модификацию.

## Hardware риски

### OpenWrt compatibility

Не каждый роутер стабильно поддерживается OpenWrt. Даже если модель есть в списке совместимости, могут быть ограничения по Wi-Fi, flash-памяти, recovery-процедурам или конкретной аппаратной ревизии.

Риски:

- модель поддерживается частично;
- нужная ревизия отличается от продаваемой в магазине;
- прошивка требует нестандартной процедуры установки;
- часть функций работает хуже, чем на vendor firmware.

### Router revisions

Одна коммерческая модель может иметь несколько hardware revisions с разными чипами, flash/RAM, Wi-Fi-модулями и bootloader behavior.

Риски:

- покупатель или поставщик присылает другую ревизию;
- инструкция для одной ревизии ломает другую;
- автоматизация provisioning становится опасной без точного определения revision.

### Flash/RAM limitations

Дешевые роутеры часто имеют мало flash и RAM. Для OpenWrt, WireGuard, DNS, routing rules, monitoring и будущего agent может не хватить ресурсов.

Риски:

- не помещаются нужные пакеты;
- роутер нестабилен после установки дополнительных сервисов;
- обновления становятся невозможными без пересборки firmware image;
- высокий memory pressure вызывает деградацию Wi-Fi или VPN.

### Recovery from bad firmware

Неудачная прошивка может превратить устройство в нерабочее для конечного пользователя.

Риски:

- нет простого recovery mode;
- recovery требует UART, TFTP или ручного вмешательства;
- поддержка не сможет помочь удаленно;
- партия устройств может стать коммерчески непригодной.

### Wi-Fi performance

Даже если VPN работает, пользователь оценивает продукт по скорости и стабильности Wi-Fi.

Риски:

- слабые драйверы Wi-Fi в OpenWrt;
- падение скорости на 2.4 GHz или 5 GHz;
- проблемы с mesh/repeater-сценариями;
- жалобы пользователя воспринимаются как проблема VPN-сервиса.

### CPU performance with VPN

WireGuard быстрый, но все равно требует CPU. На слабых роутерах VPN throughput может быть ниже ожиданий.

Риски:

- скорость VPN ниже тарифа пользователя;
- высокая нагрузка CPU ухудшает Wi-Fi и routing;
- дешевые модели не выдерживают несколько клиентов или high-bandwidth usage;
- fallback-протоколы вроде OpenVPN могут быть заметно медленнее WireGuard.

## ISP риски

### DHCP, PPPoE, static IP

У разных провайдеров разные способы подключения.

Риски:

- PPPoE требует логин/пароль и снижает производительность на слабом CPU;
- static IP требует ручной настройки;
- DHCP может конфликтовать с существующим роутером пользователя;
- нестандартные VLAN-настройки провайдера усложняют установку.

### CG-NAT

CG-NAT обычно не мешает исходящему VPN-подключению, но усложняет диагностику и некоторые входящие сценарии.

Риски:

- пользователь не понимает, почему нельзя подключиться к домашней сети извне;
- сложнее диагностировать NAT-related проблемы;
- может сочетаться с нестабильным UDP behavior.

### Double NAT

Если VPN-роутер ставится за существующим роутером провайдера, появляется double NAT.

Риски:

- проблемы с некоторыми приложениями;
- сложность объяснения пользователю схемы подключения;
- конфликты подсетей;
- нестабильность при неправильном DHCP setup.

### UDP blocking/throttling

WireGuard работает поверх UDP. Некоторые сети могут блокировать или ограничивать UDP.

Риски:

- VPN не подключается;
- скорость резко падает в конкретной сети;
- проблема выглядит как ошибка нашего роутера;
- нужен fallback-протокол или альтернативный порт.

### DNS issues

DNS — частая причина жалоб: сайты не открываются, split-routing работает странно, провайдер подменяет ответы.

Риски:

- DNS leak;
- конфликт между DNS провайдера, роутера и VPN;
- неправильная маршрутизация доменов;
- блокировки на уровне DNS.

### WireGuard fingerprinting/blocking

Plain WireGuard может быть распознан или заблокирован в некоторых сетях.

Риски:

- VPN перестает работать у части пользователей;
- сервис становится зависимым от конкретного протокола;
- требуется поддержка fallback-протоколов или маскировки.

### Need for fallback protocols like AmneziaWG/OpenVPN

Fallback-протоколы нужны не для усложнения MVP, а для снижения риска блокировок и несовместимости.

Риски:

- AmneziaWG или OpenVPN требуют дополнительных пакетов и ресурсов;
- OpenVPN может быть медленным на слабом железе;
- сложнее support и provisioning;
- архитектура должна позволять заменять provider/protocol без переписывания продукта.

## Стратегия тестирования

### Test matrix by router model

Для каждой модели и ревизии нужно фиксировать:

- model name;
- hardware revision;
- CPU/RAM/flash;
- OpenWrt version;
- firmware installation method;
- recovery method;
- WireGuard throughput;
- Wi-Fi throughput без VPN;
- Wi-Fi throughput с VPN;
- стабильность под нагрузкой;
- известные ограничения.

### Test matrix by ISP connection type

Для каждого типа подключения нужно проверить:

- DHCP;
- PPPoE;
- static IP;
- CG-NAT;
- double NAT;
- UDP behavior;
- DNS behavior;
- типичные сценарии пользователя: streaming, messenger, browser, mobile devices, smart TV.

### Minimum acceptable checklist перед продажей роутера

Перед продажей конкретной модели нужно подтвердить:

- OpenWrt устанавливается воспроизводимо;
- есть понятная recovery-процедура;
- хватает flash/RAM для текущего и ближайшего будущего набора пакетов;
- VPN throughput соответствует обещанию продукта;
- Wi-Fi performance приемлем для целевого пользователя;
- роутер стабилен после reboot и power loss;
- provisioning можно выполнить без ручной инженерной работы;
- есть инструкция поддержки для типовых ISP-сценариев;
- известны ограничения и они отражены в sales/support материалах.

## Action items

- Составить список 3-5 приоритетных моделей роутеров для тестов.
- Для каждой модели определить hardware revisions, которые реально продаются.
- Собрать baseline speed tests без VPN и с WireGuard.
- Проверить PPPoE performance отдельно от DHCP.
- Проверить сценарий роутер-за-роутером.
- Определить минимальный fallback plan при UDP/WireGuard blocking.
- Создать таблицу совместимости перед началом масштабных продаж.

## Вывод

Hardware и ISP compatibility — это продуктовые риски, а не второстепенные технические детали. Их нужно валидировать до масштабирования продаж, иначе backend и подписочная модель могут быть готовы, но пользовательский опыт будет нестабильным.
