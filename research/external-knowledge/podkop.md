# Podkop как внешний источник инженерного опыта

Podkop рассматривается как инженерный reference для маршрутизации, DNS, split-tunneling и интеграции VPN на роутере. Это не продукт для копирования, а источник идей и вопросов для анализа.

## Правило работы с внешними материалами

Сторонние прошивки, скрипты, пакеты, бинарные файлы, изображения, тексты, конфигурации и другие assets можно использовать как источники знаний и ссылок, но нельзя копировать в репозиторий, пока не проверены лицензия, условия использования и явное разрешение на хранение или модификацию.

## Что важно понять

### Routing

Для VPN-роутера важно не только поднять туннель, но и правильно направить трафик.

Нужно изучить:

- policy-based routing;
- routing отдельных подсетей;
- routing по доменам или спискам;
- поведение после reconnect;
- конфликты с локальной сетью пользователя.

### DNS

DNS часто определяет, работает ли split-tunneling так, как ожидает пользователь.

Нужно изучить:

- DNS forwarding;
- DNS leak prevention;
- доменные списки;
- взаимодействие DNS и routing rules;
- поведение при недоступности upstream DNS.

### Split-tunneling

Split-tunneling может быть нужен для сценариев, где весь трафик через VPN слишком медленный или ломает локальные сервисы.

Нужно изучить:

- routing только выбранных доменов или IP ranges;
- исключения для локальных сетей;
- UX для включения или отключения split mode;
- риски неправильной маршрутизации.

### VPN integration

Важно понять, как на роутере надежно соединяются VPN-туннель, firewall, DNS и routing.

Нужно изучить:

- порядок применения настроек;
- поведение при падении туннеля;
- kill switch;
- обновление конфигурации без поломки сети;
- диагностику для поддержки.

## Продуктовый принцип

Наш продукт должен скрывать эту сложность от конечного пользователя.

Пользователь не должен понимать:

- routing tables;
- firewall zones;
- DNS forwarding;
- nftables/iptables;
- domain lists;
- особенности VPN-протоколов.

Пользовательский UX должен быть ближе к модели:

- роутер включен;
- подписка активна;
- нужный режим выбран;
- интернет работает;
- если есть проблема, поддержка видит диагностический статус.

## Что нельзя копировать

- исходный код без проверки лицензии;
- скрипты установки;
- пакеты;
- assets;
- документацию дословно;
- готовые конфигурации без понимания прав и последствий.

## Предположения

- Podkop может подсветить типовые сложности router-side networking.
- Для MVP не обязательно реализовывать весь split-tunneling.
- Архитектура должна оставить место для router agent или provisioning logic, которые смогут управлять DNS/routing позже.
