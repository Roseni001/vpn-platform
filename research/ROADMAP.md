# Roadmap проекта vpn-platform

Этот roadmap описывает развитие продукта: платформы для продажи преднастроенных VPN-роутеров с подпиской. Документ фиксирует этапы, ожидаемые результаты и критерии готовности.

## Правило работы с внешними материалами

Сторонние прошивки, скрипты, пакеты, бинарные файлы, изображения, тексты, конфигурации и другие assets можно использовать как источники знаний и ссылок, но нельзя копировать в репозиторий, пока не проверены лицензия, условия использования и явное разрешение на хранение или модификацию.

## Stage 0: Research started

### Цель

Собрать первичное понимание рынка, похожих решений, технических подходов и рисков.

### Ожидаемый результат

В репозитории есть research-раздел с заметками по конкурентам, железу, OpenWrt, WG-Easy и идеям продукта.

### Критерии завершения

- Созданы первые research-документы.
- Зафиксированы ключевые направления: роутеры, прошивки, VPN-протоколы, провайдерские ограничения.
- Появилось базовое понимание, что продуктовый риск связан не только с backend, но и с железом, прошивкой и ISP.

## Stage 1: Infrastructure with VPS, Docker, wg-easy

### Цель

Поднять минимальную VPN-инфраструктуру, на которой можно выдавать WireGuard-доступы и проверять поведение клиентов.

### Ожидаемый результат

Есть VPS-инфраструктура с Docker и wg-easy как первичным VPN-provider backend.

### Критерии завершения

- VPS доступен и управляем.
- Docker-инфраструктура описана или подготовлена.
- wg-easy поднят и может создавать WireGuard-клиентов.
- Понятны базовые API-возможности wg-easy для интеграции с платформой.

## Stage 2: Backend foundation with VPNProvider abstraction

### Цель

Создать backend-основу, не привязанную напрямую к WG-Easy.

### Ожидаемый результат

В backend есть FastAPI-приложение, интерфейс `VPNProvider`, реализация `WGEasyProvider`, сервисный слой и первые endpoints для управления VPN-клиентами.

### Критерии завершения

- API работает через сервисы и интерфейсы.
- Endpoints не знают о WG-Easy.
- Используется async `httpx`.
- Есть централизованная конфигурация, logging и обработка ошибок.
- Нет базы данных, authentication и Docker-логики внутри backend MVP.

## Stage 2.1: Domain model documented

### Цель

Зафиксировать основную доменную модель продукта до добавления базы данных и бизнес-логики.

### Ожидаемый результат

Создан документ `research/domain-model.md`, описывающий сущности `Device`, `User`, `VPNClient`, `Subscription`, `ActivationToken`, `VPNServer` и связи между ними.

### Критерии завершения

- Описаны основные сущности и причины их существования.
- Зафиксировано, что VPN-provider является заменяемой инфраструктурной зависимостью.
- Подготовлена база для проектирования Device Registry и Subscription logic.

## Stage 3: Device Registry

### Цель

Создать реестр физических роутеров, которые продает и обслуживает платформа.

### Ожидаемый результат

Появляется доменная и техническая модель устройства: serial number, модель, revision, firmware version, статус, владелец, дата активации.

### Критерии завершения

- Есть модель `Device`.
- Есть статусы жизненного цикла устройства.
- Можно зарегистрировать устройство до продажи.
- Можно привязать устройство к пользователю.
- Устройство отделено от VPN-клиента и подписки.

## Stage 4: Router Activation / Provisioning

### Цель

Создать безопасный процесс активации роутера и выдачи VPN-конфигурации.

### Ожидаемый результат

Пользователь или роутер может пройти activation flow через `ActivationToken`, после чего система создает или назначает VPN-доступ.

### Критерии завершения

- Есть модель `ActivationToken`.
- Токен нельзя использовать повторно без разрешенного сценария.
- Токен имеет срок действия и статус.
- После активации устройство получает VPNClient или ссылку на конфигурацию.
- Ошибки активации логируются и понятны для поддержки.

## Stage 5: Subscription logic

### Цель

Связать право пользоваться VPN-сервисом с оплатой и тарифом.

### Ожидаемый результат

Появляется модель `Subscription`, которая управляет доступом устройств к VPN.

### Критерии завершения

- Есть статусы подписки: trial, active, past_due, expired, cancelled.
- Истекшая или неоплаченная подписка может ограничить или отключить VPNClient.
- Подписка не смешана с Device и VPNClient.
- Есть базовая стратегия продления, отмены и grace period.

## Stage 6: Admin panel

### Цель

Дать оператору или поддержке инструмент для управления пользователями, устройствами, подписками и VPN-доступом.

### Ожидаемый результат

Появляется административный интерфейс или API, через который можно просматривать и исправлять состояние системы.

### Критерии завершения

- Можно найти пользователя, устройство и VPNClient.
- Можно видеть статус подписки и активации.
- Можно переиздать VPN-конфигурацию.
- Можно отключить или заменить устройство.
- Все опасные действия аудитируются.

## Stage 7: OpenWrt router agent

### Цель

Создать минимальный агент или provisioning-механику на стороне роутера, чтобы скрыть сложность настройки VPN от пользователя.

### Ожидаемый результат

Роутер умеет получить конфигурацию, применить ее и сообщить платформе базовый статус.

### Критерии завершения

- Определен безопасный способ первичной активации.
- Роутер может применить VPN-конфигурацию без ручных действий пользователя.
- Есть базовый health/status сигнал от устройства.
- Есть стратегия обновления конфигурации.
- Нет копирования сторонних прошивок или пакетов без проверки лицензий.

## Stage 8: Hardware/ISP testing

### Цель

Проверить, что выбранные роутеры и VPN-подход работают на реальных типах подключений и у разных интернет-провайдеров.

### Ожидаемый результат

Сформирована матрица совместимости по моделям роутеров, ревизиям, прошивкам и типам ISP-подключений.

### Критерии завершения

- Проверены ключевые модели роутеров.
- Проверены DHCP, PPPoE, static IP, CG-NAT и double NAT сценарии.
- Проверены UDP-блокировки и деградация WireGuard.
- Определены fallback-протоколы или ограничения продаж.
- Есть минимальный checklist перед продажей роутера.

## Общий принцип

Backend, provisioning и router-side logic должны развиваться независимо. Продуктовая модель должна оставаться устойчивой даже при замене WG-Easy, WireGuard, конкретной модели роутера или способа прошивки.
