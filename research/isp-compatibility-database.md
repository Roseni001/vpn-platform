# База ISP-совместимости

Этот документ описывает план сбора статистики совместимости интернет-провайдеров, роутеров, прошивок и VPN-профилей.

Важно: блокировка VPN-протоколов — движущаяся цель. Нельзя утверждать, что какой-либо протокол является неблокируемым. Результат зависит от ISP, региона, DPI-правил, портов, профиля трафика, типа подключения и времени.

## Цель

Цель — построить собственную knowledge base о том, какие router models, firmware versions, protocols, ports и ISPs работают надежно.

Эта база нужна не только для аналитики. Она должна стать практическим инструментом для:

- выбора рекомендуемого protocol profile;
- поддержки пользователей;
- принятия решений о закупке роутеров;
- планирования fallback-протоколов;
- выявления рискованных ISP и регионов.

## Почему это стратегически важно

Платформа продает управляемый VPN-роутер, поэтому пользователь ожидает, что устройство будет работать в его сети. Но реальные условия отличаются:

- разные провайдеры;
- разные регионы;
- DHCP, PPPoE, static IP, mobile;
- CG-NAT и double NAT;
- UDP blocking/throttling;
- DNS-подмена или DNS-блокировки;
- разные модели роутеров и hardware revisions.

Без собственной базы совместимости команда будет решать проблемы реактивно. С базой совместимости можно заранее предупреждать риски и улучшать рекомендации.

## Предлагаемые поля

Минимальная структура записи:

- `provider_name`
- `region`
- `connection_type`: DHCP / PPPoE / static IP / mobile / unknown
- `router_model`
- `router_revision`
- `firmware_version`
- `protocol_profile`
- `server_region`
- `connection_result`: success / failed / unstable / unknown
- `failure_reason`
- `latency_ms`
- `download_speed_mbps`
- `upload_speed_mbps`
- `packet_loss_percent`
- `dns_issues`
- `udp_blocked`
- `notes`
- `tested_at`

В будущем можно добавить:

- anonymized device class;
- firmware build channel;
- ISP ASN;
- port;
- transport;
- retry count;
- last known working profile;
- support ticket reference без персональных данных.

## Как данные будут собираться

### Ручное лабораторное тестирование

Лабораторные тесты нужны до продаж и перед добавлением новых моделей роутеров.

Что проверять:

- конкретную модель и revision;
- firmware version;
- WireGuard profile;
- fallback profiles;
- DHCP/PPPoE/static IP;
- throughput;
- DNS behavior;
- reboot/power loss behavior.

### Кейсы поддержки

Каждое обращение в поддержку может дать сигнал о совместимости.

Важно фиксировать:

- провайдера и регион;
- тип подключения;
- модель роутера;
- firmware version;
- какой protocol profile был активен;
- что помогло решить проблему;
- какой профиль оказался рабочим.

### Будущая телеметрия router agent

В будущем router agent может отправлять техническую телеметрию.

Примеры:

- connection attempts;
- selected protocol profile;
- success/failure;
- latency;
- packet loss;
- DNS status;
- active server region;
- firmware version.

Телеметрия должна быть минимальной и технической. Не нужно собирать содержимое трафика, посещенные сайты или лишние персональные данные.

### Диагностика при первичной активации клиента

Во время первичной активации можно собрать диагностический минимум:

- тип подключения, если определяем автоматически;
- успешность первого VPN-подключения;
- выбранный protocol profile;
- server region;
- базовая network health информация.

Эти данные помогут улучшать onboarding и рекомендации без ручного опроса пользователя.

## Как данные будут использоваться

### Выбор рекомендуемого protocol profile

Если для конкретного ISP/region/router model известно, что WireGuard нестабилен, backend сможет рекомендовать fallback profile.

На раннем этапе это должно быть ручное или простое rule-based решение, а не сложная автоматическая система.

### Улучшение диагностики в поддержке

Support сможет быстрее отвечать:

- какие проблемы типичны для этого ISP;
- какой protocol profile пробовать;
- нужна ли смена server region;
- есть ли известная проблема с моделью роутера или firmware.

### Выбор роутеров для продажи

Если модель показывает плохую производительность, сложный recovery или нестабильность на популярных ISP, ее не стоит продавать массово.

### Приоритизация протоколов

Если данные показывают частые проблемы с UDP/WireGuard, fallback-протоколы получают больший приоритет. Если проблемы редкие, можно не усложнять MVP.

### Выявление рискованных ISP и регионов

База позволит отмечать регионы и провайдеров, где требуется осторожный sales/support подход:

- предупреждать о возможных ограничениях;
- рекомендовать другой профиль;
- рекомендовать другой router model;
- временно не продавать в проблемный сегмент без тестов.

## Примечание о приватности

Не хранить лишние персональные данные. Хранить только технические connectivity data, необходимые для диагностики и улучшения продукта.

Не нужно хранить:

- содержимое пользовательского трафика;
- посещенные сайты;
- private keys;
- полные IP-адреса без необходимости;
- персональные данные без явного бизнес-основания.

Если позже потребуется связывать запись с пользователем или устройством, нужно использовать минимально необходимую ссылку и продумать retention policy.

## Качество данных

Данные совместимости должны иметь источник и уровень доверия.

Примеры источников:

- lab_test;
- support_case;
- router_agent;
- onboarding_diagnostic;
- operator_note.

Примеры confidence:

- low: единичный сигнал;
- medium: несколько повторений;
- high: воспроизводимый lab test или много подтверждений.

## Вывод

ISP compatibility data — стратегический актив платформы. Она поможет не только чинить проблемы, но и принимать продуктовые решения: какие роутеры продавать, какие протоколы поддерживать, какие регионы масштабировать и как снижать риск плохого пользовательского опыта.
